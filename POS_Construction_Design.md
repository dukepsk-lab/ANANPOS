# ระบบ POS & หลังบ้าน สำหรับร้านวัสดุก่อสร้าง (1 สาขา, ธุรกิจครอบครัว)
### System Design Document — Version 1.0

---

## 0. ภาพรวมและหลักการออกแบบ (Design Principles)

ก่อนลงรายละเอียด ขอวางหลักคิด 4 ข้อที่จะกำกับทุกการตัดสินใจ เพราะนี่คือ "ธุรกิจครอบครัว 1 สาขา" ไม่ใช่ Enterprise — over-engineer เมื่อไหร่คือพังเมื่อนั้น:

1. **Reliability > Features** — POS ห้ามล่ม ห้ามขายไม่ได้ ต่อให้เน็ตหลุด ระบบขายหน้าร้านต้องเดินต่อได้ ดังนั้นเราจะเลือกสถาปัตยกรรมที่ทำงาน "ในร้าน" เป็นหลัก
2. **UI ต้องโง่ที่สุดเท่าที่จะโง่ได้** — ปุ่มใหญ่ ตัวหนังสือใหญ่ ขั้นตอนน้อย แม่/ป้า/น้าใช้ได้โดยไม่ต้องอบรม การขาย 1 บิลควรจบใน 3-4 จิ้ม
3. **ดูแลคนเดียวได้** — คุณคือคนดูแล ดังนั้นเลือก stack ภาษาเดียว เครื่องมือที่คุณคุ้น (TypeScript ecosystem) ไม่ต้องสลับไปมาหลายภาษา
4. **เริ่มเล็ก สเกลทีหลัง** — ออกแบบ schema ให้รองรับอนาคต (หลายสาขา, หลายราคา) แต่ "เปิดใช้" เฉพาะที่จำเป็นวันนี้

## 1. System Architecture & Tech Stack

### 1.1 Stack ที่แนะนำ (ตรงกับสิ่งที่คุณถนัดอยู่แล้ว)

| ชั้น (Layer) | เทคโนโลยี | เหตุผล |
|---|---|---|
| **Frontend + Backend** | **Next.js (App Router) + TypeScript** | Full-stack ในตัวเดียว เขียน UI กับ API ในโปรเจกต์เดียว ภาษาเดียว — คุณใช้ทำ SWINGBOX มาแล้ว ลดภาระการดูแล |
| **ORM / Data Layer** | **Prisma** | Type-safe, migration ง่าย, คุณคุ้นมือ — สำคัญมากสำหรับ schema ที่ซับซ้อนแบบหน่วยนับหลายหน่วย |
| **Database** | **PostgreSQL** | เสถียร รองรับ transaction หลายคนพร้อมกัน (แคชเชียร์ + คนหลังบ้าน) มี audit/backup ดี — เผื่อสเกลหลายสาขาในอนาคต |
| **UI Components** | **Tailwind CSS + shadcn/ui** | สร้าง UI เรียบ ปุ่มใหญ่ ได้เร็ว ปรับขนาดตัวอักษรง่าย |
| **Print** | **ESC/POS (react-thermal-printer)** สำหรับใบเสร็จ + **HTML→PDF** สำหรับใบกำกับภาษี A4 | ใบเสร็จออกจากเครื่องพิมพ์ความร้อน, ใบกำกับภาษีออก A4 |
| **Process Manager** | **PM2** หรือ **Docker Compose** | รัน app ให้เด้งกลับเองเวลาเครื่องรีสตาร์ท |

> **หมายเหตุเรื่อง DB:** ถ้าอยากเริ่มเบาที่สุดและมีคนใช้พร้อมกันไม่เกิน 1-2 คนจริงๆ SQLite ก็พอ (เหมือนที่คุณใช้ใน SWINGBOX) แต่ร้านวัสดุมักมีคนขายหน้าร้าน + คนคิดเงิน + คนดูสต็อกพร้อมกัน ผมแนะนำ **PostgreSQL** ตั้งแต่แรกเลย จะได้ไม่ต้องย้ายทีหลัง การ migrate Prisma จาก SQLite→Postgres มี edge case น่ารำคาญ

### 1.2 รูปแบบการ Deploy — นี่คือการตัดสินใจที่สำคัญที่สุด

ร้านวัสดุก่อสร้างมักอยู่พื้นที่เน็ตไม่นิ่ง ผมแนะนำ **On-Premise เป็นหลัก + Cloud Backup**:

```
┌─────────────────────────────────────────────┐
│         ในร้าน (Local Network / LAN)          │
│                                               │
│   ┌──────────────┐      ┌─────────────────┐  │
│   │ Mini PC       │◄────►│ แท็บเล็ต/จอสัมผัส │  │
│   │ (เครื่อง Server)│      │ (หน้าขาย POS)    │  │
│   │ - Next.js app │      └─────────────────┘  │
│   │ - PostgreSQL  │      ┌─────────────────┐  │
│   │ - รัน 24 ชม.   │◄────►│ PC หลังบ้าน      │  │
│   └──────┬───────┘      │ (สต็อก/รายงาน)   │  │
│          │              └─────────────────┘  │
│          │ + เครื่องพิมพ์ใบเสร็จ + ลิ้นชักเงิน    │
└──────────┼────────────────────────────────────┘
           │ Backup อัตโนมัติทุกคืน
           ▼
   ☁️ Cloud (Google Drive / S3) — สำรองฐานข้อมูล
```

**ข้อดี:** เน็ตหลุดก็ขายได้ ตอบสนองเร็ว (ไม่ต้องวิ่งผ่านอินเทอร์เน็ต) ค่าใช้จ่ายต่อเดือนเกือบ 0

**ทางเลือก B (Cloud/VPS):** ถ้าเน็ตร้านนิ่งมากและอยากเข้าถึงจากที่ไหนก็ได้ → เช่า VPS ถูกๆ (~150-300 บาท/เดือน) แต่ **ข้อเสียคือเน็ตหลุด = ขายไม่ได้** ซึ่งรับไม่ได้สำหรับ POS หน้าร้าน

### 1.3 ฮาร์ดแวร์ที่ต้องเตรียม (ต้นทุนประมาณ)

- **Mini PC** (เช่น Intel N100 mini PC) เป็น server: ~5,000-9,000 บาท
- **จอสัมผัส / แท็บเล็ต** หน้าขาย: ~3,000-8,000 บาท
- **เครื่องพิมพ์ใบเสร็จความร้อน 80mm**: ~1,500-3,000 บาท
- **เครื่องพิมพ์ A4** สำหรับใบกำกับภาษี (หรือ Dot Matrix ถ้าต้องการสำเนา/กระดาษก๊อปปี้): มีอยู่แล้วน่าจะพอ
- **ลิ้นชักเก็บเงิน (Cash Drawer)** ต่อกับเครื่องพิมพ์: ~1,000-2,000 บาท
- **เครื่องสแกนบาร์โค้ด** *(ออปชัน)*: ~800-2,000 บาท

> ⚠️ **สำคัญ:** วัสดุก่อสร้างส่วนใหญ่ **ไม่มีบาร์โค้ด** (ปูน เหล็ก หิน ทราย) ดังนั้น UI ต้องเน้น **ค้นหาด้วยชื่อ/เลือกจากหมวด** ให้เร็ว ส่วนบาร์โค้ดทำไว้สำหรับสินค้าที่อยากสแกน (เช่น สี อุปกรณ์ปลีก) โดยพิมพ์ฉลากบาร์โค้ดเอง

## 2. Database Schema

ผมจัดกลุ่มตารางตามโดเมน ตารางที่เป็น "หัวใจและซับซ้อนที่สุด" คือกลุ่มหน่วยนับ (UOM) จะอธิบายละเอียดเป็นพิเศษ

### 2.1 กลุ่มสินค้า & หน่วยนับ (Product & UOM) — ส่วนที่ยากที่สุด

**แนวคิดหลัก:** สินค้าแต่ละตัวมี **หน่วยฐาน (base unit)** ใช้เก็บสต็อก 1 หน่วยเดียวเสมอ แล้วผูก **หน่วยขายได้หลายหน่วย** เข้ากับมัน พร้อม "อัตราแปลง" และ "ราคาต่อหน่วย" แยกกัน

> ตัวอย่าง: ปูน base unit = **ถุง** → ขายเป็น "ถุง" (1:1) หรือ "พาเลท" (1 พาเลท = 50 ถุง) ก็ได้ เวลาขาย 1 พาเลท ระบบหักสต็อก 50 ถุงอัตโนมัติ และ **ราคา/พาเลทถูกกว่าราคา 50 ถุงปลีก** (เพราะซื้อเหมา) — จึงต้องตั้งราคาแยกต่อหน่วย ไม่ใช่คูณเอา

```prisma
model Product {
  id            Int            @id @default(autoincrement())
  code          String         @unique          // รหัสสินค้า
  name          String                          // ชื่อสินค้า เช่น "ปูนทีพีไอ เขียว"
  categoryId    Int
  baseUnitId    Int                             // หน่วยฐาน (ใช้เก็บสต็อกเสมอ)
  reorderPoint  Float          @default(0)       // จุดสั่งซื้อ (ต่ำกว่านี้ = เตือนเติมสต็อก)
  isStockItem   Boolean        @default(true)    // false = สินค้าไม่นับสต็อก (สั่งทำ/เหมา)
  isActive      Boolean        @default(true)
  productUnits  ProductUnit[]
}

model Unit {
  id            Int      @id @default(autoincrement())
  name          String   @unique                // ถุง, พาเลท, เส้น, มัด, คิว, ตัน, อัน, ม้วน
  abbreviation  String?
}

model ProductUnit {
  id              Int      @id @default(autoincrement())
  productId       Int
  unitId          Int
  conversionRate  Float                          // 1 หน่วยนี้ = ? หน่วยฐาน (เช่น พาเลท=50)
  price           Float                          // ราคาขายต่อหน่วยนี้ (ตั้งแยก!)
  isDefaultSale   Boolean  @default(false)        // หน่วยขายเริ่มต้น
  barcode         String?  @unique
  @@unique([productId, unitId])
}
```

**เคสพิเศษที่ schema นี้รองรับ:**
- **เหล็ก** เส้น/มัด → 1 มัด RB6 = 100 เส้น (แต่ RB9 อาจ = 50 เส้น) เก็บ conversionRate รายสินค้า ไม่ใช่ค่ากลาง ✅
- **หิน/ทราย** คิว/ตัน → อัตราแปลงคิว↔ตัน **ขึ้นกับชนิดวัสดุและความชื้น** จึงเก็บแยกรายสินค้าได้เลย หรือถ้าขาย "เหมากอง" ให้ใช้ flag `isStockItem=false` แล้วกรอกจำนวน/ราคามือ ✅

### 2.2 กลุ่มสต็อก (Inventory)

ใช้หลัก **Stock Movement Ledger** (บันทึกการเคลื่อนไหวทุกครั้ง) เพื่อ audit ได้ ว่าของหายไปไหน + คำนวณต้นทุน (COGS) ได้

```
StockMovement
├─ id
├─ productId
├─ movementType   // PURCHASE_IN | SALE_OUT | RETURN_IN | ADJUST | DAMAGE_OUT
├─ quantityBase   // จำนวนในหน่วยฐาน (+/-)
├─ unitCost       // ต้นทุนต่อหน่วย (ไว้คำนวณกำไร/มูลค่าสต็อก)
├─ refType        // SALE | PURCHASE | RETURN | ADJUSTMENT
├─ refId          // อ้างถึงเอกสารต้นทาง
├─ note           // เหตุผล (โดยเฉพาะ ADJUST/DAMAGE)
├─ createdBy
└─ createdAt

StockBalance  (แคชยอดคงเหลือ — อัปเดตทุกครั้งที่มี movement เพื่อความเร็ว)
├─ productId  (PK)
└─ quantityOnHand   // หน่วยฐาน
```

### 2.3 กลุ่มลูกค้า & เครดิต (Customer & Credit)

```
Customer
├─ id
├─ name
├─ phone, address
├─ type            // CASH (ขาจร) | CREDIT (เครดิต/ช่างขาประจำ)
├─ creditLimit     // วงเงินเครดิต
├─ creditTermDays  // เทอม เช่น 30 วัน
├─ priceTier       // ระดับราคา (เผื่อให้ราคาพิเศษช่างขาประจำ) *ดู Edge Case #10*
├─ taxId           // เลขผู้เสียภาษี (ออกใบกำกับภาษี)
└─ balance         // ยอดค้างปัจจุบัน (แคช)
```

### 2.4 กลุ่มการขาย (POS / Sales)

```
Sale  (บิลขาย 1 ใบ)
├─ id
├─ billNo            // เลขบิล (running)
├─ customerId        // null = ขาจรเงินสด
├─ saleDate
├─ subtotal
├─ discountAmount    // ส่วนลดท้ายบิล ✅
├─ vatAmount
├─ grandTotal
├─ paymentType       // CASH | CREDIT | PARTIAL
├─ paidAmount
├─ status            // PAID | UNPAID | PARTIAL | VOID
├─ docType           // RECEIPT (ใบเสร็จ) | TAX_INVOICE (ใบกำกับภาษี)
└─ createdBy

SaleItem  (รายการในบิล)
├─ id
├─ saleId
├─ productId
├─ productUnitId     // ขายด้วยหน่วยไหน (ปูน=พาเลท?)
├─ quantity          // จำนวนตามหน่วยที่ขาย
├─ unitPrice
├─ discount          // ส่วนลดต่อรายการ (ถ้ามี)
└─ lineTotal
```

### 2.5 กลุ่มลูกหนี้ / วางบิล / รับชำระ (AR)

```
Invoice  (ใบวางบิล — รวมหลายบิลเครดิตมาเรียกเก็บทีเดียว)
├─ id
├─ invoiceNo
├─ customerId
├─ invoiceDate, dueDate
├─ totalAmount
└─ status            // OPEN | PARTIAL | PAID

InvoiceSale  (junction: 1 ใบวางบิล ↔ หลายบิลขาย)
├─ invoiceId
└─ saleId

Payment  (การรับชำระ / ผ่อนชำระ)
├─ id
├─ paymentNo
├─ customerId
├─ invoiceId         // null ได้ ถ้าจ่ายตรงเข้าบิล
├─ amount
├─ method            // CASH | TRANSFER | CHEQUE
├─ paymentDate
└─ note
```
> 💡 **การผ่อนชำระ** = 1 Invoice มีได้หลาย Payment จนยอดครบ → status เปลี่ยน OPEN → PARTIAL → PAID. การจัดสรรเงินที่จ่ายมาควรเป็น FIFO (ตัดบิลเก่าก่อน) *ดู Edge Case #7*

### 2.6 กลุ่มจัดส่ง (Delivery)

Delivery

### 2.7 กลุ่มจัดซื้อ & ตีคืน & ผู้ใช้ (Purchase / Return / User)

```
Supplier { id, name, contact, ... }

Purchase { id, supplierId, purchaseDate, total, status }
PurchaseItem { id, purchaseId, productId, productUnitId, quantity, unitCost }

Return  (ใบตีคืนสินค้า)
├─ id, returnNo

ReturnItem

```

## 3. UI Screens / Wireframe Flow

จัดกลุ่มตาม "ความถี่การใช้งาน" — เมนูที่ใช้ทุกวันต้องอยู่หน้าแรก ปุ่มใหญ่สุด

### 🟢 กลุ่มใช้ทุกวัน (Daily — ปุ่มใหญ่บนหน้าหลัก)

| # | หน้าจอ | จุดสำคัญด้าน UX |
|---|---|---|
| 1 | **Login (เลือกชื่อ + PIN 4 หลัก)** | ไม่ต้องพิมพ์รหัสยาว แตะรูป/ชื่อตัวเอง ใส่ PIN จบ |
| 2 | **หน้าหลัก (Dashboard)** | ปุ่มใหญ่ 6 ปุ่ม: ขายของ / ลูกหนี้ / จัดส่ง / สต็อก / รายงาน / ตั้งค่า + แสดงยอดขายวันนี้ตัวโตๆ |
| 3 | **หน้าขายของ (POS)** ⭐ | ช่องค้นหาใหญ่บนสุด → แตะสินค้า → **เด้งถามหน่วย** (ถุง/พาเลท) → ใส่จำนวน → เพิ่มลงบิล. ด้านขวาเป็นตะกร้า มีปุ่ม "ส่วนลดท้ายบิล" ตัวโต และปุ่ม "เก็บเงิน" สีเขียวใหญ่ |
| 4 | **หน้าชำระเงิน (Payment)** | เลือก เงินสด/เงินเชื่อ → ถ้าเงินสด: ใส่เงินรับ คำนวณเงินทอนตัวโต → เลือก พิมพ์ใบเสร็จ/ใบกำกับภาษี |

**Flow การขาย 1 บิล (เป้าหมาย ≤4 จิ้ม):**
```
ค้นหา/แตะสินค้า → เลือกหน่วย+จำนวน → [ทำซ้ำ] → กดเก็บเงิน → เลือกเงินสด/เครดิต → พิมพ์ → จบ
```

| 5 | **ลูกหนี้/เงินเชื่อ (AR)** | ลิสต์ลูกหนี้ค้างชำระ (เรียงตามค้างนานสุด) → กดเข้าดูประวัติ → **วางบิล** / **รับชำระ-ผ่อน** |
| 6 | **รายละเอียดลูกค้า** | ข้อมูลติดต่อ, วงเงิน, ยอดค้าง, ประวัติซื้อ/จ่าย |
| 7 | **คิวจัดส่ง (Delivery Board)** | บอร์ดแบบ Kanban 3 คอลัมน์: **รอส่ง \| กำลังส่ง \| สำเร็จ** ลากการ์ดเปลี่ยนสถานะ หรือแตะปุ่มเปลี่ยนสถานะ |
| 8 | **สต็อกสินค้า (Inventory)** | ลิสต์สินค้า + ยอดคงเหลือ, ไฮไลต์ **สีแดงของใกล้หมด**, ปุ่ม "ปรับสต็อก/นับสต็อก", ปุ่ม "ของเสีย" |
| 9 | **รับสินค้าเข้า (Stock In / Purchase)** | บันทึกของเข้าจาก Supplier → เพิ่มสต็อก + บันทึกต้นทุน |
| 10 | **ตีคืนสินค้า (Return)** | เลือกบิลเดิม → เลือกรายการคืน → ระบุ คืนเข้าสต็อก/ของเสีย → คืนเงินสด/ลดหนี้ |

### 🔵 กลุ่มดูเป็นครั้งคราว (รายงาน + ตั้งค่า)

| # | หน้าจอ | หน้าที่ |
|---|---|---|
| 11 | **รายงานยอดขายรายวัน** | สรุปยอดวันนี้/เลือกช่วงวัน, แยกเงินสด/เครดิต, สินค้าขายดี |
| 12 | **รายงานสินค้าคงคลัง** | ของคงเหลือ + รายการ "ควรสั่งซื้อเติม" (ต่ำกว่า reorder point) → กดส่งออกเป็นใบสั่งซื้อได้ |
| 13 | **รายงานลูกหนี้ค้างชำระ (Aging)** | แยกตามอายุหนี้ (0-30, 31-60, 60+ วัน) |
| 14 | **ตั้งค่าสินค้า** | เพิ่ม/แก้สินค้า, **ตั้งหน่วยนับหลายหน่วย + ราคาแต่ละหน่วย**, จุดสั่งซื้อ |
| 15 | **ตั้งค่าระบบ** | ข้อมูลร้าน (หัวบิล), ผู้ใช้, หน่วยนับ, **สำรองข้อมูล** |

> 🎯 **เคล็ดลับ UX สำหรับครอบครัวทุกวัย:** ปรับขนาดฟอนต์ได้ใน Settings (ตัวเล็ก/กลาง/ใหญ่), ใช้ไอคอน + คำไทยคู่กันทุกปุ่ม, สีสถานะชัด (เขียว=ดี แดง=ต้องสนใจ), ทุกการลบ/ยกเลิกต้องมี "ยืนยันอีกครั้ง" กันจิ้มพลาด

## 4. Edge Cases — ปัญหาที่ร้านวัสดุเจอจริง + วิธีให้ระบบจัดการ

ส่วนนี้คือสิ่งที่ระบบสำเร็จรูปทั่วไปทำพลาด แต่ร้านวัสดุเจอทุกวัน:

### 4.1 การตีคืนสินค้า (Return) — *เคสที่ 1*
ลูกค้าซื้อปูน 50 ถุง ใช้ไม่หมด คืน 10 ถุง
- **ระบบควร:** สร้างใบ Return อ้างบิลเดิม → ถาม "คืนเข้าสต็อกหรือของเสีย?" → ถ้าเข้าสต็อก: `StockMovement: RETURN_IN +10 ถุง` → คืนเงินสด **หรือ** ออก Credit Note ลดยอดลูกหนี้ (ถ้าซื้อเชื่อ)
- **กับดัก:** ถ้าบิลนั้นวางบิลไปแล้ว การคืนต้องไปแก้ยอด Invoice ด้วย — ระบบต้องเช็กว่าบิลถูกผูกใน Invoice ไหนหรือยัง

### 4.2 สินค้าเสียหาย/เสื่อมสภาพ (Damage) — *เคสที่ 2*
ปูนแข็ง (โดนความชื้น), เหล็กขึ้นสนิม, กระเบื้องแตก, สีหมดอายุ
- **ระบบควร:** ปุ่ม "ตัดของเสีย" ในหน้าสต็อก → `StockMovement: DAMAGE_OUT` พร้อม **บังคับใส่เหตุผล** → เข้ารายงานต่างหากเพื่อดูว่าเสียหายเดือนละเท่าไหร่ (เป็นต้นทุนแฝงที่ร้านวัสดุมักไม่รู้ตัว)

### 4.3 หน่วยนับแปลงไม่ลงตัว / ขายแยกจากแพ็ค — *เคสที่ 3*
ซื้อปูนมาเป็นพาเลท แต่ขายปลีกเป็นถุง / เปิดพาเลทขายเหลือเศษ
- **ระบบควร:** เก็บสต็อกเป็น **หน่วยฐาน (ถุง) เสมอ** การซื้อพาเลทเข้า = +50 ถุง, ขายปลีก = -1 ถุง → ไม่มีปัญหาเศษ เพราะทุกอย่างวิ่งที่หน่วยฐาน

### 4.4 คิว ↔ ตัน ไม่คงที่ (หิน/ทราย) — *เคสที่ 3b*
ทราย 1 คิว หนักไม่เท่ากันตามความชื้น/ชนิด
- **ระบบควร:** เก็บ conversionRate **รายสินค้า** (ทรายหยาบ ≠ ทรายละเอียด ≠ หิน) และอนุญาตให้ **แก้จำนวน/น้ำหนักตอนขายได้** เพราะของจริงชั่งแล้วไม่ตรงเป๊ะ

### 4.5 ขายเหมากอง (Lump-sum) — *เคสที่ 4*
"ทรายกองนี้ เหมา 3,000" ไม่ได้วัดคิวเป๊ะ
- **ระบบควร:** ตั้งสินค้าแบบ `isStockItem=false` หรือมีโหมด "ขายเหมา" ที่กรอกราคามือ + หักสต็อกประมาณการด้วยมือ ไม่ต้องบังคับให้ระบบคำนวณคิว

### 4.6 ของส่งไม่ครบ/ส่งหลายเที่ยว (Partial Delivery) — *เคสที่ 5*
สั่งอิฐ 5,000 ก้อน รถบรรทุกได้ทีละ 2,000 ส่ง 3 เที่ยว
- **ระบบควร:** 1 Sale ผูกได้หลาย Delivery แต่ละเที่ยวมีสถานะแยก → บิลปิดเมื่อส่งครบ

### 4.7 ลูกหนี้จ่ายไม่ครบ/ทยอยจ่าย — *เคสที่ 7*
ค้าง 50,000 จ่ายมา 20,000
- **ระบบควร:** บันทึก Payment 20,000 → ตัด **บิลเก่าสุดก่อน (FIFO)** → ยอดค้างเหลือ 30,000 → Invoice status = PARTIAL → ประวัติแสดงทุกงวดที่จ่าย

### 4.8 ราคาผันผวน (เหล็ก/ปูนขึ้นราคาบ่อย) — *เคสที่ 9*
- **ระบบควร:** เก็บ **ประวัติราคา** และที่สำคัญคือ **เก็บราคา ณ วันขายไว้ใน SaleItem** (ไม่ดึงราคาปัจจุบันมาแสดงย้อนหลัง) เพื่อให้บิลเก่าไม่เพี้ยนเมื่อราคาเปลี่ยน. ต้นทุนใช้ moving average จาก unitCost ใน StockMovement

### 4.9 ราคาหลายระดับ (ช่างขาประจำได้ราคาพิเศษ) — *เคสที่ 10*
ขาจรราคาหนึ่ง, ช่างประจำอีกราคาหนึ่ง
- **ระบบควร:** ใช้ field `priceTier` ใน Customer → ตอนเลือกลูกค้าในบิล ระบบดึงราคาตาม tier อัตโนมัติ (เริ่มแรกทำแค่ 2 ระดับ: ปลีก/ช่าง ก็พอ)

### 4.10 ยกเลิกบิลหลังพิมพ์/หักสต็อกแล้ว (Void) — *เคสที่ 11*
- **ระบบควร:** ไม่ลบบิลทิ้ง (ห้ามลบ เพื่อ audit) แต่เปลี่ยน status = VOID → **กลับรายการ StockMovement อัตโนมัติ** (คืนของเข้าสต็อก) + ถ้าเป็นเครดิตก็ลดยอดลูกหนี้คืน → บันทึกว่าใครยกเลิก เมื่อไหร่ ทำไม

### 4.11 สต็อกติดลบ (ขายก่อนบันทึกของเข้า) — *เคสที่ 12*
ของมาถึงแล้วแต่ยังไม่ได้คีย์รับเข้า ลูกค้ามารอซื้อ
- **ระบบควร:** ตั้งค่าได้ว่าจะ "เตือนแต่ขายได้" (แนะนำสำหรับร้านวัสดุ เพราะของจริงมีอยู่) ไม่ใช่ "บล็อกห้ามขาย" — แล้วค่อยไปคีย์รับเข้าทีหลัง สต็อกจะกลับมาบวก

### 4.12 ตัดเหล็ก/ตัดของตามขนาด — *เคสที่ 14*
ลูกค้าขอเหล็กยาว 3 เมตร จากเส้น 10 เมตร
- **ระบบควร:** รองรับขายเศษ (จำนวนทศนิยม) หรือทำเป็นสินค้า "บริการตัด" แยก + คิดราคาตามเมตร

### 4.13 เก็บเงินปลายทาง (COD) — *เคสที่ 16*
คนขับรถไปเก็บเงินตอนส่ง
- **ระบบควร:** บิลตั้งสถานะ "ส่งแล้วรอเก็บเงิน" → เมื่อคนขับกลับมา บันทึก Payment ปิดยอด → flag `isCOD` ใน Delivery ช่วยให้รู้ว่าเที่ยวไหนต้องถือเงินกลับมา

## 📌 ภาคผนวก: แผนพัฒนาแบบเป็นเฟส (แนะนำลำดับการทำ)

อย่าทำทุกอย่างพร้อมกัน — ทำให้ขายได้ก่อน แล้วค่อยต่อยอด:

**Phase 1 (MVP — ขายได้จริง):** สินค้า+หน่วยนับ → POS ขายเงินสด → พิมพ์ใบเสร็จ → สต็อกตัดอัตโนมัติ → รายงานยอดขายรายวัน
*เป้าหมาย: เลิกใช้สมุด/เครื่องคิดเลข*

**Phase 2 (เครดิต):** ลูกค้าเครดิต → ขายเชื่อ → ลูกหนี้ → วางบิล → รับชำระ/ผ่อน → รายงาน Aging
*เป้าหมาย: เลิกจดหนี้ในสมุด*

**Phase 3 (ครบวงจร):** จัดส่ง Kanban → ตีคืน/ของเสีย → รับสินค้าเข้า/Supplier → ใบกำกับภาษี → ราคาหลายระดับ

**Phase 4 (สเกล):** หลายราคา/โปรโมชัน → ออนไลน์ดูรายงานจากมือถือ → (อนาคต) หลายสาขา
